# 業務文書作成スキル

## このスキルの目的

ユーザー（ゆういちさん）の職場向け文書——委員会資料、報告資料、議事録、案内・依頼文など——を、渡された材料（メモ、箇条書き、口頭説明の書き起こし）から実用レベルに仕上げる。目指すのは「読み手が短時間で理解し、判断・行動できる文書」。飾った長文ではなく、簡潔で誤読されない文書を作る。

## 文書作成の前に確認すること

不明なら最初にまとめて確認する（質問攻めにしない）：

1. **読み手と目的** — 誰が読むか（委員会メンバー/上司/全社員）、読んだ人に何をしてほしいか（承認/判断/周知/記録）。
2. **文書の位置づけ** — 審議してもらう資料か、報告だけか。これで構成が変わる。
3. **分量・様式の指定** — A4一枚か、枚数自由か。社内テンプレートの有無（あれば構成やサンプルを見せてもらい、それに合わせる）。
4. **提出日と会議日** — 日付・回次などの記載に必要。

※ユーザーの職場固有の様式・慣行がわかったら、このスキルに追記して型を固定することを提案する。

## 文書共通の原則

- **結論先出し**：1行目〜冒頭段落で「何の件で、要は何か」がわかるようにする。経緯から書き始めない。
- **事実と意見・提案を区別する**：「〜であった（事実）」と「〜すべきと考える（意見）」を混ぜない。報告資料の信頼性はここで決まる。
- **数字は具体的に**：「多数」「近日中」ではなく「12件」「7月15日まで」。
- **一文を短く**：一文一義。「〜であり、〜だが、〜なので」と続く文は分割する。
- **読み手の知識に合わせる**：委員会に非技術系メンバーがいるなら、電気・設備の専門用語には短い注釈を付ける。
- **A4一枚に収める工夫**：詳細は別紙・添付に逃がし、本紙は判断に必要な情報だけにする。

## 文書タイプ別の型

### 委員会資料（審議・報告事項）
```
1. 件名（「〜について（審議/報告）」の別を明記）
2. 目的・背景（2〜3行）
3. 内容（審議なら：案の内容と選択肢・比較／報告なら：結果と数値）
4. 審議いただきたい点（審議事項のみ。何を決めてほしいかを明文化）
5. 今後の予定
6. 添付資料
```
審議資料では「決めてほしいこと」が曖昧なまま出すのが最大の失敗パターン。ここを必ず明確にする。

### 報告資料・報告書
```
1. 件名・報告者・日付
2. 結論/要旨（3行以内）
3. 経緯・実施内容（時系列は表も活用）
4. 結果（数値・事実）
5. 課題・所見（事実と区別して）
6. 今後の対応（担当・期限つきで）
```
不具合・事故系の報告では「発生状況 → 応急処置 → 原因 → 恒久対策 → 水平展開」の流れを使う。原因が未確定の段階では「調査中」と正直に書き、推定は推定と明記する。

### 議事録
- 冒頭に：日時・場所・出席者・議題。
- 発言の逐語記録ではなく「**決定事項・宿題（担当と期限）・継続審議**」の3分類を主役にする。議論の経過は要点のみ。
- 録音の文字起こしを渡された場合はtoken-saverの圧縮方針を併用し、上記3分類に整理する。

### 案内文・依頼文
- 「いつ・どこで・誰が・何を・どうしてほしいか」を落とさない。依頼文は相手の手間（何をどこまでやればよいか、所要時間、期限）を明示すると回収率が上がる。

## 成果物の形式

- 正式な提出文書 → Word（docxスキル併用）。社内様式があるならそれに合わせたレイアウトで。
- 会議で映す資料 → PowerPointも可（pptxスキル併用）。
- 集計・一覧を含む資料 → 表はExcel別紙（xlsxスキル併用）として分けることも提案する。
- 構成の相談や短い文書はチャット内で完結してよい。すべてをファイル化しない。

## 推敲を頼まれたときの観点

既存文書の添削では、(1)結論が冒頭にあるか、(2)事実と意見が混在していないか、(3)読み手が次に取るべき行動が明確か、(4)冗長表現（「〜させていただく」の乱発、二重敬語、重複）、(5)数字と日付の具体性、の順で見る。直した箇所には理由を一言添え、ユーザーの文体を尊重して直しすぎない。
