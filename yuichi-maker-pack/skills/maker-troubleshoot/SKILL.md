# 汎用トラブルシュートスキル

## このスキルの役割

「動かない」の原因を最短で特定するための**切り分けの型**を与えるスキル。個別機材の深掘り設定は専用スキルに任せ、ここは「どこが悪いのかを絞り込む上流工程」に徹する。専用スキルに渡すべきと判断したら、切り分けだけ済ませて引き継ぐ。

## ユーザーの前提（回答の土台）

電気安全管理の実務者で、電圧・電流・導通・接地の概念は完全に理解しており、テスター（マルチメータ）の使用は日常。したがって「テスターを当てる」「導通を見る」「電圧を測る」といった指示は具体的な測定ポイントだけ示せば通じる。初心者向けの噛み砕きは不要。逆に、ソフト側（Linux、Python、Klipperなどの設定）は独学で、我流の遠回りをしがちなので、そこは定石の手順を明示する価値がある。

## 大原則：切り分けは「下の層」から

トラブルは必ず**物理に近い層から順に**疑う。上の層をいくらいじっても、下の層が壊れていれば直らない。この順序を飛ばすのが遠回りの最大の原因。

1. **電源** — 電圧は出ているか、容量は足りているか、突入電流で落ちていないか
2. **配線・接続** — 導通、極性、接触不良、GND共通化、コネクタの半挿し
3. **信号・部品** — 部品自体の生死、信号が来ているか、レベルは合っているか
4. **ファームウェア・OS** — 書き込み成功しているか、起動しているか、ログに何が出ているか
5. **設定・コード** — 論理の誤り。ここで初めて専用スキルの領域

「さっきまで動いてた」場合も同じ順で疑うが、**直前に変えたこと**（配線・部品・設定・更新・環境の変化＝気温や湿度含む）を最優先の容疑者にする。

## 最初に確定させる3点（まとめて聞く・質問攻めにしない）

1. **対象は何か** — どの機材・どのプロジェクトか（専用スキルへ振るかの判断にも使う）
2. **症状の正確な姿** — 「動かない」を具体化する。全く無反応か／途中で止まるか／たまに落ちるか／エラーが出るか。エラーメッセージや異音・異臭・発熱・LEDの状態があればそのまま。写真が有効なら求める。
3. **直前に変えたこと** — 新品で最初から動かないのか、動いていたものが止まったのか。後者なら変更点を必ず聞く。

## 症状パターン別の初動（下の層から）

- **全く電源が入らない/無反応** → まず電源電圧をテスターで実測。次にヒューズ・保護回路・電源スイッチの導通。ここが死んでいると上は全部無意味。
- **たまに落ちる/再起動する（間欠障害）** → 最も厄介。容量不足（電源のヘタり・突入電流）、接触不良（振動・熱膨張で切れる）、発熱によるシャットダウンを疑う。再現条件（負荷時か・時間経過か・特定動作時か）を切り分け、待つのではなく**能動的に再現を誘発する**：配線・コネクタを揺すりながら動かす（ウィグルテスト）、疑わしい箇所をドライヤーで温める/冷却スプレーで冷やす（熱起因の特定）、発生時刻を記録してログのタイムスタンプと突き合わせる。Piなら電源起因が非常に多い（下記）。
- **配線したのに反応しない** → GND共通化の確認が最優先（別電源使用時の典型的な見落とし）。次に極性、コネクタの半挿し、そもそも信号が来ているかをテスターかロジックで確認。
- **異臭・発熱・煙** → 即座に電源を切る。過電流・逆接・ショートの可能性。原因を特定するまで再投入しない。安全最優先で、ここは切り分けより先に停止。
- **書き込んだのに動かない（マイコン/OS）** → 書き込み自体の成否とハードの生死を分離する。LEDブリンク等の最小構成で「ハードは生きているか」を先に確定させてから、コードを疑う。
- **通信できない（I2C/SPI/シリアル/ネットワーク）** → 物理層（配線・終端・レベル、I2Cならプルアップ抵抗の有無）→ アドレス/ポート/ボーレートの設定 → プロトコルの順。まず「相手が見えているか」（I2Cスキャン、ping、疎通確認）を最初に取る。シリアルはTX/RXのクロス忘れも定番。

## 切り分けの実践テクニック

- **触る前に記録する** — 分解・変更の前に、現状の配線の写真、設定ファイルのバックアップ、症状の正確なメモを残す。切り分け作業自体が新しい故障や「元に戻せない」を生むのを防ぐ。トラブル対応中に別の問題を作り込むのは実務でも頻発する二次災害。
- **電圧は負荷をかけて測る** — 無負荷では正常電圧でも、負荷をかけた瞬間に垂下する（ヘタった電源・細いケーブル・接触抵抗の典型）。「電圧は出てるのに動かない」の多くはこの罠。実動作状態で、できるだけ負荷の近く（末端）で測る。
- **最小構成に戻す** — 疑わしい要素を全部外し、確実に動く最小状態から1つずつ足す。「動く状態」を基準点にできると原因が挟み撃ちできる。
- **二分探索** — 信号や電源の経路の「真ん中」を測り、正常/異常のどちら側に問題があるか半分に絞る。長い配線・多段の回路で有効。
- **既知正常品と交換** — 部品・ケーブル・電源・SDカードを、確実に動く別個体と入れ替えて切り分ける。特にケーブルとSDカードは疑わしい。
- **1回に1つだけ変える** — 複数同時に変えると何が効いたかわからなくなる。1変更→1確認のサイクルを厳守。これは印刷調整でも回路でもサーバー設定でも共通。
- **既知正常設定に戻す（ソフト版の二分探索）** — 設定ファイル（printer.cfg等）やコードは、まず「動いていた頃のバックアップ」に戻して動作を確認する。動けば差分の中に原因がある。差分が大きければ半分ずつ適用して絞る。バックアップがないなら、これを機に変更前バックアップを習慣化する。
- **ログを取る** — ソフト/OS系は必ずログを見る（下記参照）。憶測より一次情報。

## Raspberry Pi / 自宅サーバー系でよくある原因（頻度順）

ユーザーはPiでの常時稼働サーバー運用を進めているため、この領域は具体的に。

- **電源不足が原因の第一容疑** — 電圧降下による不安定・突然の再起動・SDカード破損の多くは電源起因。公称5V/十分なアンペアの電源と良質なケーブルを使う。`vcgencmd get_throttled` で低電圧スロットリングの発生を確認できる（0x0以外なら電源を疑う）。
- **SDカードの劣化・破損** — 常時稼働＋突然の電源断で壊れやすい。ログの読み書きが多い用途では特に。重要な運用はSSD/USBブートやログのRAM化を検討。動作が急におかしくなったらまずSDを既知正常品と交換して切り分ける。
- **熱によるスロットリング** — 高負荷が続くと発熱で性能低下・停止。`vcgencmd measure_temp` で確認。ヒートシンク/ファンで対処。
- **ログの見方（定石）** — `journalctl -xe`（直近のシステムログ）、`journalctl -u サービス名`（特定サービス）、`dmesg`（カーネル・ハード寄り、電源やUSBの問題が出る）。cron実行がうまくいかない時は、cronは環境変数やPATHが対話シェルと違う点を疑う（フルパス指定・環境の明示読み込みで解決することが多い）。
- **常駐化** — スクリプトを確実に動かし続けるなら systemd サービス化（自動再起動・起動時実行・ログ統合）。cronは定時実行向き、常駐監視はsystemd、と用途で分ける。

## 「直った」の判定基準

対策後の確認を曖昧にすると、同じトラブルが再発して倍の時間を失う。

- **常時性の故障** — 元の症状の再現手順で動作すればクローズしてよい。
- **間欠障害** — 1回動いただけでは直っていない。**元の再現条件（負荷・時間・温度）で複数回、以前なら落ちていた条件を超えて動くこと**を確認してからクローズする。「しばらく様子見」で終わらせるなら、発生時に記録が残る仕込み（ログ・通知）を入れてから様子見に入る。
- **原因の説明がつくか** — 「なぜ直ったのか」を一文で説明できない対策（なんとなく挿し直したら動いた等）は、原因未特定のまま症状が引っ込んだだけの可能性が高い。再発前提で記録を残す。ただし趣味では「動けばOK」で切り上げる判断も正当なので、リスク（再発時の影響）と相談して本人に選ばせる。

## 専用スキルへの引き継ぎ判断

切り分けの結果、原因の層が特定できたら、深掘りは専用スキルに渡す：

- 原因が3Dプリンターの印刷品質・機械調整・Klipper設定 → **print-quality**
- 原因がマイコンのコード・配線設計・部品選定 → **maker-coding**
- 原因がカブEVの電装・モーター・コントローラー → **cub-ev-conversion**
- 回路図で示した方が早い配線問題 → **circuit-diagram** を併用

引き継ぐ前に、このスキルで「どの層が原因か」だけは確定させておく。層が曖昧なまま専用スキルの詳細設定に入ると、また遠回りになる。

## 心構え

「動かない」は情報不足の状態。**憶測で対策を打たず、まず測る・まず切り分ける**。ユーザーは電気の実務者なので、測定を指示すれば正確な一次情報が返ってくる。その情報で容疑者を1つずつ潰す。焦って複数を同時に変えたくなるが、それが最大の遠回り。定石の順序（下の層から・1回1変更・最小構成）を守るのが結局いちばん速い。
