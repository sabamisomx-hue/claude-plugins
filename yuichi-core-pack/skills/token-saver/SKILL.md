# トークン節約スキル

## なぜ重要か

トークンは会話の「容量」であり、コストでもある。冗長な前置き・繰り返し・過剰な要約もどきの長文は、読む側の時間とAPIコストの両方を無駄にする。特に以下のような場面で効果が大きい：

- 決算説明会・IR動画・YouTube投資系動画の文字起こしを何本も分析するとき（元の文字起こし自体が数千〜数万字になりがち）
- 複数銘柄・複数レポートを横断的に比較するとき（会話が長くなるほど、過去の文脈を圧迫する）
- 単純な質問に対して長すぎる回答が返ってくると、本当に重要な情報が埋もれる

ただし、簡潔さは正確さやニュアンスを犠牲にしてよい理由にはならない。投資判断に関わる前提条件・リスク・不確実性は、たとえ短くても落とさない。

## モード1: 回答そのものを簡潔にする

- 結論から書く。背景説明や「良い質問ですね」のような前置きは省く。
- 質問文をそのまま繰り返さない。ユーザーはすでに自分が何を聞いたか知っている。
- すでに会話で共有済みの前提（保有銘柄、投資方針など）を毎回説明し直さない。
- 箇条書き・表を使い、同じ内容を散文でも繰り返さない（例：表で数値を示した後に、文章でもう一度同じ数値を説明しない）。
- 「〜と考えられます」「〜と言えるでしょう」のような曖昧なクッション表現を多用しない。断定できることは断定し、不確実な部分だけ明示的に「不確実」と書く。
- 選択肢や比較を長文で説明するより、表・箇条書きの方が短く正確に伝わる場面ではそちらを使う。
- 例外：医療・法律・投資の最終判断に関わる免責事項や、誤解すると損害につながる注意書きは、短くても必ず残す。「簡潔にする」は「重要な警告を消す」ことではない。

## モード2: 長文（文字起こし・レポート・記事）の圧縮

長文をそのまま全部読み込んで逐語的に要約すると、出力もまた長くなりがちで、トークンの節約にならない。以下の手順で「情報密度を上げる」ことを意識する。

1. **目的を先に確認する** — 「この文字起こしの何を知りたいか」（銘柄の業績評価か、市場全体の見立てか、単なる記録保存か）によって、拾うべき情報が変わる。目的が不明な場合は、全文要約ではなく「結論・数値・リスク要因」を優先するデフォルト方針を取る。
2. **雑談・相槌・重複発言は切り捨てる** — 「えー」「そうですね」のようなフィラー、同じ趣旨の言い換えの繰り返しは要約に含めない。
3. **数値・固有名詞・意思決定に関わる発言を優先的に残す** — 業績数値、ガイダンス、銘柄名、日付、具体的なリスク要因などは削らない。定性的な感想・一般論は大胆に削ってよい。
4. **定型フォーマットで出力する**（下記テンプレート参照）— 散文でだらだら書くより、決まった型に流し込む方が短く、比較もしやすい。
5. **原文の言い回しをそのまま引用しない** — 自分の言葉で圧縮する。逐語引用は著作権の観点からも避け、必要な場合でも短い一節に留める。
6. **省略した部分があることを明示する** — 「詳細な質疑応答部分は割愛」のように、何を削ったかを一言添えることで、ユーザーが必要なら原文に戻れるようにする。

### 出力テンプレート例（長文要約時）

```
## 結論（1〜3行）
## 主要な数値・事実
- 箇条書きで数値や固有名詞を中心に
## リスク・不確実な点
- 断定できない部分はここに集約
## （必要なら）次のアクション
```

このテンプレートは固定ではなく、対象が「決算説明会」「マクロ経済ニュース」「単なる議事録」かによって見出しを調整してよい。重要なのは「結論を先頭に置き、数値と定性情報を分離する」という考え方。

## 判断の目安

- 明示的に「簡潔に」「要約して」「トークン節約して」と言われたとき → 積極的にモード1・2を適用する。
- 長い文字起こしやレポートが貼り付けられ、分析・要約を求められたとき → 依頼されなくてもモード2の圧縮方針を自動適用する。
- 逆に、ユーザーが「詳しく」「全部知りたい」「深掘りして」と言った場合は、このスキルより詳細さを優先する。トークン節約は目的であって、情報を隠す口実にしてはいけない。

## ir-watchとの連携

決算説明会・株主総会・IRイベントの文字起こしを渡されたら、このスキルの圧縮方針で処理した上で、収集の文脈（どの銘柄の何の決算か、持ち越し論点）はir-watchスキルの型に沿って整理する。圧縮で落としてはいけない情報: 通期予想の修正、配当方針、経営陣の定性的なトーン変化（強気/慎重の転換）。
