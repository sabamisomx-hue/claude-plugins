# 構造化リサーチ・サマリー（Research-Simplifier）

## 目的

ユーザーが入力した長文テキスト、論文、Web記事から、**意思決定に必要な核心的情報だけを短時間で把握できる**ように構造化要約する。読み物としての要約ではなく、「次の判断・行動につなげるための要約」を作る。

## 出力フォーマット（4セクション固定）

必ず以下の4つのセクションで出力する。順番・見出しを変えない。

```
1. 【結論】この情報が伝える最も重要なポイント（3行以内）
2. 【核心的なファクト】記載されている重要な数値、固有名詞、データ（箇条書き）
3. 【背景と理由】なぜその結論になるのかのロジック（箇条書き）
4. 【ユーザーへの示唆】この情報から次にどんなアクションやリサーチが必要か（2〜3点）
```

## 制約事項（厳守）

- **専門用語の平易化**：専門用語は中学生でも理解できる一般的な言葉に置き換えるか、初出時に短い注釈を付ける。ただし置き換えによって意味が変わる場合（法令用語、規格名、医学用語など）は原語を残して注釈で説明する。
- **事実と意見の分離**：元データにない憶測・推測を混ぜない。原文の中でも、著者の主張（Opinion）と裏付けデータ（Fact)を区別して扱う。【核心的なファクト】には検証可能な事実だけを置き、著者の見解は【背景と理由】側で「著者は〜と主張」の形で書く。
- **【ユーザーへの示唆】だけは例外的にClaudeの提案を含む**が、それが原文由来ではなくClaudeの提案であることが読み手に分かる書き方をする。
- 原文の言い回しの長い引用はしない。自分の言葉で圧縮する。

## 運用上の注意

- 原文に結論が明示されていない場合（データの羅列、途中経過の報告など）は、【結論】に「原文には明確な結論がない」ことを正直に書き、読み取れる範囲の要点を示す。無理に結論をでっち上げない。
- 複数の記事・論文を一度に渡された場合は、1件ずつこのフォーマットで出し、最後に横断的な比較を短く添える。
- 数値を要約に載せるときは、単位・期間・条件（例：前年比か前期比か）を落とさない。条件を落とした数値は誤解のもとになる。
- URLだけ渡された場合はweb_fetchで本文を取得してから要約する。取得できない場合は要約できない旨を伝える（想像で埋めない）。
- ユーザーの意思決定の文脈（投資判断、業務判断、購入検討など）が会話からわかる場合、【ユーザーへの示唆】はその文脈に合わせて具体化する。

## ir-watch・投資系スキルとの連携

決算資料・IRレポート・業界調査の要約では、【ユーザーへの示唆】セクションを投資判断の文脈（保有銘柄への影響、購入理由との整合、ir-watchのバイアス対策=悪材料を先に）に合わせて書く。銘柄の深掘りが続くならstock-analysisへの引き継ぎを提案する。
