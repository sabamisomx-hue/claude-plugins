# プロジェクト立ち上げスキル

## このスキルの役割

構想（ふわっとした「作りたい」）を、**着手可能な最初の一歩**まで落とし込む。ゴールは立派な計画書ではなく、「今日これをやればいい」が明確になった状態。ユーザーは反復改善型なので、長大な事前計画は作らない。

## 整理の型（この順で埋める）

会話しながら以下を埋めていく。全部を質問攻めで聞かず、会話から読み取れるものは埋めた上で、足りない要点だけ確認する。

### 1. 完成の定義（最重要）
「何ができたら完成か」を一文で。ここが曖昧なプロジェクトは必ず漂流する。
- 例:「スマホから冷蔵庫の中身を入れるとレシピが返ってくる」「文化祭で膳所城とN-ゲージが同じ台の上に載っている」
- **誰が使うのか**も同時に確定する（自分だけ／家族／不特定多数）。これで技術選定と作り込みの深さが大きく変わる。

### 2. 制約の確認
- **期限** — 文化祭・決算日のような固定期限があるか、ないか。固定期限があれば逆算式、なければ反復式で計画する。
- **予算** — 上限額。既存の手持ち機材・部品で流用できるものを先に棚卸しする（ユーザーは機材が豊富なので、買う前に持ち物確認が定石）。
- **時間** — 平日夜/週末どれくらい割けるか。
- **技術的な未知** — やったことがない要素はどれか。未知の要素が2つ以上重なる工程は事故りやすいので、分離して個別に検証する計画にする。

### 3. 方式の選定
複数のやり方があるときは、2〜3案を「手間・費用・仕上がり・学び」で短く比較して提示し、ユーザーに選ばせる。このとき:
- ユーザーの既存スキル・機材で戦える案を優先する（Fusion 360、3Dプリンター2台＋MSLA、電気工事の実務知識、Python/Klipper経験）
- 「買うか作るか」で迷う要素が出たら **buy-or-diy** スキルの判断軸を使う
- 新技術の習得自体が目的の一部なら、遠回りでも学びの多い案を否定しない（趣味プロジェクトの価値は完成物だけではない）

### 4. 工程の分解と「最初の動くもの」
- 全体を工程に割るが、詳細化するのは直近の工程だけ。先の工程は見出しレベルで留める（どうせ変わる）。
- **最初のマイルストーンは「最小の動くもの」に置く** — 全機能の1割でも、端から端まで通ることを優先する。例: レシピアプリなら「固定の材料リストで1レシピ返る」まで、EV化なら「スタンド上で後輪が回る」まで。
- 未知の要素・リスクの高い要素は**前倒しで検証**する（後回しにすると手戻りが最大化する）。

### 5. 最初の一歩
「今日・今週末にやること」を1つに絞って具体的に示す。買い物が必要ならその型番・数量まで。

## 出力の形式

整理結果はこの程度のコンパクトさでまとめる（長文の企画書にしない）:

- **完成の定義**: 一文
- **使う人**: 自分用/配布 など
- **期限・予算**: あれば
- **方式**: 選定結果と理由を1〜2行
- **工程**: 箇条書き4〜6項目、直近だけ詳しく
- **リスク（未知要素）**: 先に検証すべきもの
- **最初の一歩**: 今日やること1つ

## ユーザーの傾向を踏まえた注意

- **構想が広がりやすい** — 会話中に機能や欲しいものが増えていったら、「まず動く最小構成」と「後で足す拡張」に仕分けして、最初の計画には最小構成だけ入れる。拡張案は捨てずにリスト化して残す。
- **並行プロジェクトが多い** — 新規着手時に、既存の走っているプロジェクト（EV化・ジオラマ・サーバー等）との時間の取り合いになりそうなら一言確認する。ただし説教はしない。趣味は並行するものである。
- **投資の判断軸との共通性** — ユーザーは投資で「買った理由が達成されたら売る。理由がすり替わったら要注意」という軸を持つ。プロジェクトでも同様に「作る理由」を最初に言語化しておくと、途中で目的がすり替わったとき（手段の目的化）に本人が気づける。完成の定義はその錨になる。

## 他スキルとの連携

- 方式選定で買う/作るの判断 → **buy-or-diy**
- 着手後の設計詳細 → **fusion360-design**（CAD）、**maker-coding**（マイコン）、**cub-ev-conversion**（カブ）等の専用スキル
- サーバー系の構築 → **linux-server-ops**
- 業務系の文書化が必要 → **gyomu-bunsho**
